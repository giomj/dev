## J â€” n2yo.com

The rate limits are fine for the read patterns a single-device "what's above me" client actually needs, but they are the wrong shape for a shared backend: 1000/hr on `/tle/{id}` and `/positions/{id}/...` is generous per-object, but 100/hr on `/visualpasses`, `/radiopasses`, and `/above` is the binding constraint the moment more than one user or more than a handful of watched satellites hits the same key (`sources/n2yo/endpoints.md`). A backend serving N users each polling passes for even 2-3 satellites a few times a day will blow through 100/hr well before N reaches double digits, and there's no documented paid tier or key-pooling mechanism in the catalog to fall back to, so the practical fallback is architectural, not commercial: request passes rarely and serve them from a local cache, and give each *device* its own key rather than proxying all users through one shared server key. TLEs should be pulled on a scheduled cadence (a few times a day per watched object, well under the 1000/hr ceiling even for a moderate satellite list) and stored server-side; passes and positions should then be computed client-side or server-side with a local SGP4 implementation seeded from the cached TLE, which converts an API call budget problem into a compute problem we fully control. This also buys resilience: if the key is rate-limited or revoked, the app degrades to "propagate from the last-known TLE" instead of hard-failing, with a staleness warning once the TLE exceeds ~2-3 days old (LEO drift becomes visually noticeable past that). Concretely: cache TLEs keyed by NORAD ID with a fetched-at timestamp, run SGP4 locally (e.g. via `python-sgp4` or a vetted JS/Kotlin port) for all pass/position math, and reserve live n2yo `/visualpasses`/`/radiopasses` calls only for a periodic reconciliation check rather than per-user-request â€” this is the only design that scales past a single user on the free tier.

## K â€” SatDump

A real ground-station build is: a directional or omni LEO antenna (QFH or turnstile for 137 MHz NOAA/Meteor APT/LRPT, or a helical/patch for 1.7 GHz HRPT/AHRPT L-band imagery) â†’ a low-noise amplifier mounted *at the antenna* (not at the radio) to overcome cable loss before the noise floor sets in â†’ coax to the SDR â†’ host machine running the pipeline. For radio choice, an RTL-SDR (RTL-SDR Blog v3/v4) is the right entry point for 137 MHz weather-sat APT/LRPT â€” it's the cheapest device in SatDump's support matrix and fully adequate at that frequency (`sources/satdump/catalog.md`); if the goal extends to L-band AHRPT/HRPT (MetOp, NOAA-20, FengYun) an Airspy Mini or AirspyHF+ is the better call because of superior front-end linearity and a wider usable bandwidth, and both are supported on desktop *and* Android. HackRF and LimeSDR Mini buy transmit capability and wider tuning range that a receive-only weather-sat station doesn't need, so they're over-spec for this use case; BladeRF and the Pluto/AD9361 family should be avoided if any Android use is planned since neither is supported there. On host platform, Docker is the right choice for a fixed, unattended, reproducible ground station (headless recording + pipeline execution, easy to redeploy, plays well with cron-scheduled AOS/LOS pass capture) while Android (F-Droid build) is the right choice for a portable/field rig but comes with the real cost of a reduced SDR set (no BladeRF, no Pluto) and no headless scheduling â€” it's a manual, one-pass-at-a-time tool there. The genuine engineering-competence bar in SatDump is not the GUI happy path (pick pipeline, hit Start) but three places: (1) RF chain design â€” antenna polarization/gain, LNA placement and its noise-figure math, feedline loss budget, and gain/bias-tee settings passed via `--gain`/`--bias`, all of which require understanding link budgets, not just software config; (2) pipeline/parameter selection â€” matching `--samplerate`, `--baseband_format`, and the correct pipeline ID to the actual satellite/instrument, where a wrong sample rate or format silently produces garbage rather than an error; and (3) build-from-source on Linux/RPi (the recommended path on several platforms), including the RPi3-and-older `make -j1` memory workaround â€” this is a real systems-engineering step, not a package-manager install (`sources/satdump/catalog.md`).

## L â€” GOGPT

The request-form CSV/XLSX distribution means there is no API to poll, so the pipeline has to treat each release as a manually-triggered, versioned import rather than a live feed: a human (or a scheduled reminder) requests the file from Global Energy Monitor's download page, drops the raw XLSX into a dated, immutable `raw/` landing zone (e.g. `sources/gogpt/raw/2026-01/`), and a transform script converts it to a normalized long-format table rather than overwriting any shared "current" table in place â€” every release must be individually retained, never mutated, because column schemas have already changed at least four times (Aug 2023 GGPTâ†’GOGPT rename adding oil coverage, then column name/addition changes in Aug 2024, Jan 2025, and Aug 2025) (`sources/gogpt/catalog.md`). The concrete data-engineering plan is a per-release **column mapping file** (e.g. `schema_map_2025-08.yml` mapping raw headers â†’ canonical internal field names) checked into version control alongside the raw file, so that a single canonical schema can be reconstructed across releases without ever needing all releases to share literal column names; any column that's genuinely new (not renamed) gets added to the canonical schema as nullable, and any renamed column gets an explicit old-nameâ†’new-name entry with the release it changed in, so lineage is auditable. Ingestion should be idempotent and content-hashed (store a SHA256 of the raw file next to it) so re-running the pipeline on an already-ingested release is a no-op, and the transform step should assert row-count and key-column (plant owner, status, coordinates, COâ‚‚ fields) presence before writing output, failing loudly rather than silently propagating a malformed parse. Because there's no API, the "ingestion" step itself is manual-in-the-loop (someone must fill the request form with name/email/purpose) â€” this should be modeled explicitly as a task with an owner and a due date tied to the known bi-annual cadence (January, July/August) rather than automated away, and the pipeline downstream of the raw drop should be the only fully automated part.

## M â€” Integration plan

The proposed `sources/{n2yo,satdump,gogpt}` + `samples/` + `notes/` layout is sound but needs three additions to be operationally real: (1) a `sources/<name>/releases/` or dated-raw convention for GOGPT specifically (per L above) since it is the only artifact here without a live endpoint and therefore the only one where "current" is ambiguous without a timestamp; (2) a `sources/n2yo/.env.example` (committed) paired with a real `.env` (gitignored) for the API key, loaded via a standard secrets pattern â€” never hardcode the key in `endpoints.md`-adjacent code, and if this ever runs in CI or a shared server, the key belongs in the platform's secret store (e.g. GitHub Actions secrets, or a vault) and is injected as an environment variable at runtime, not committed anywhere in the repo history; and (3) a top-level `LICENSES.md` or per-source `LICENSE` stub since the three sources sit on three different legal regimes â€” n2yo (proprietary ToS + rate-limited key, no redistribution of bulk pulls implied), SatDump (GPL-family, so any *bundled/modified* copy of the software itself, not just its output, inherits copyleft obligations), and GOGPT (CC BY 4.0, permissive but attribution-mandatory). For the CC BY 4.0 attribution specifically, add a lightweight CI check â€” a script or GitHub Action step that fails the build if any file under `sources/gogpt/` or any derived table lacks the required citation string `Global Oil and Gas Plant Tracker, Global Energy Monitor, <Month Year> release` in an adjacent `ATTRIBUTION.md` or embedded header, since this catalog explicitly flags that attribution strings ship inside the download and must be preserved (`sources/gogpt/catalog.md`). GOGPT re-ingestion cadence should be a recurring calendar task (not cron, since it's a manual request-form process) fired each January and July/August tied to GEM's actual release history, writing to a new dated `raw/` directory and re-running the schema-mapped transform; n2yo needs no re-ingestion cadence at all since it's live, but the local TLE cache (per J) should be refreshed on a much tighter automated schedule (hours, not months) that is entirely independent of the GOGPT cadence.
